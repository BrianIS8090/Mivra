# JS Import Plugin Design

Дата: 2026-06-01

## Цель

Добавить в Mivra внешний плагин `markitdown-import`, который конвертирует выбранный пользователем файл в Markdown, извлекает доступные изображения и вставляет результат в текущий документ.

Ключевое архитектурное решение: импортёр остаётся внешним плагином, но базовое приложение расширяет `MivraPluginApi` небольшим assets API. Этот API отдаёт плагинам существующий S3/local-assets pipeline Mivra, не перенося конвертацию документов в ядро.

Плагин ориентируется на идею Microsoft MarkItDown - превращать документы в структурированный Markdown, - но не использует Python-пакет `markitdown` как runtime-зависимость. Вместо этого первый релиз реализует нужное подмножество конвертеров на JavaScript.

## Область

Входит в первый релиз:

- внешний плагин `plugins/markitdown-import/`;
- пакет `.mivraplugin` для установки через менеджер плагинов;
- выбор файла с диска через `<input type="file">` внутри диалога плагина;
- конвертация полностью в renderer-процессе через JS-библиотеки, упакованные вместе с плагином;
- поддержка форматов:
  - `.docx`;
  - `.xlsx`;
  - `.pdf` с текстовым слоем;
  - `.txt`;
  - `.md`, `.markdown`;
  - `.csv`;
- предпросмотр полученного Markdown;
- извлечение изображений из `.docx` и best-effort обработка изображений PDF;
- сохранение извлечённых изображений через существующую систему Mivra:
  - если S3 настроен, secret сохранён и последний тест соединения прошёл, загрузить байты в S3;
  - иначе сохранить байты в локальную папку `assets/` рядом с текущим документом;
- вставка Markdown-ссылок на сохранённые изображения;
- режимы применения результата:
  - вставить в конец текущего документа;
  - заменить текущий документ;
  - вставить с заголовком и разделителем;
  - скопировать Markdown в буфер обмена без изменения документа;
- понятные ошибки для неподдержанного формата, неуспешного парсинга и пустого результата.

Не входит в первый релиз:

- запуск конвертации документов в ядре Mivra;
- запуск Python-процесса из Mivra;
- локальный HTTP-сервер;
- упаковка Python или Microsoft MarkItDown в установщик Mivra;
- импорт URL;
- OCR для сканов;
- аудио, видео, изображения, EPUB, ZIP и Outlook-сообщения;
- высокоточное восстановление сложной верстки PDF;
- точечная вставка в позицию курсора, потому что текущий Plugin API не отдаёт selection.

## Архитектура

Решение состоит из двух частей.

Первая часть - расширение базового Plugin API. В `MivraPluginApi` добавляется namespace `assets`, который позволяет доверенному плагину сохранить бинарный asset через тот же механизм, что использует ручная вставка изображений в Mivra.

Вторая часть - внешний JS-плагин. Он регистрирует пункт меню `Плагины -> Import to Markdown`, открывает диалог, читает выбранный файл как `File`, выбирает конвертер по расширению и MIME type, показывает Markdown и применяет результат через `api.document.setContent(...)`.

Конвертеры живут внутри плагина:

- `docxToMarkdown(file, assets)` читает `.docx` через браузерную JS-библиотеку для Word-документов, получает HTML или структурированный текст, сохраняет изображения через `assets.saveBytes(...)` и нормализует результат в Markdown.
- `xlsxToMarkdown(file)` читает workbook, проходит по листам и формирует Markdown-разделы с таблицами.
- `pdfToMarkdown(file, assets)` читает PDF через JS PDF-парсер, извлекает текстовый слой постранично и добавляет разделители страниц. Изображения PDF обрабатываются best-effort: если библиотека отдаёт отдельные image resources, они сохраняются через `assets.saveBytes(...)`; если нет, плагин не обещает точного восстановления вложенных картинок.
- `textToMarkdown(file)` читает `.txt`, `.md`, `.markdown` и возвращает текст с нормализацией переносов строк.
- `csvToMarkdown(file)` читает CSV и формирует Markdown-таблицу.

Все зависимости должны попадать в собранный `index.js` плагина или лежать рядом в assets плагина. Плагин не должен импортировать приватные модули Mivra и не должен рассчитывать на зависимости из `node_modules` приложения.

## Assets API

В `MivraPluginApi` добавляется:

```ts
type MivraPluginApi = {
  assets: {
    saveBytes(input: {
      bytes: Uint8Array;
      filename: string;
      alt?: string;
      kind?: 'image' | 'file';
    }): Promise<{
      url: string;
      markdown: string;
      storage: 's3' | 'local';
    }>;
  };
};
```

Для `kind: 'image'` метод возвращает Markdown вида `![alt](url)`. Для `kind: 'file'` - `[filename](url)`.

Поведение `saveBytes`:

1. Проверить permission `assets:write`.
2. Проверить имя файла и размер через существующие правила загрузки.
3. Если S3 полностью готов (`s3`, secret, `s3Verified`), вызвать существующий `s3UploadBytes(...)`.
4. Если S3 не готов, сохранить байты локально через `saveLocalAssetBytes(...)`.
5. Для локального сохранения требовать, чтобы текущий документ был сохранён и `baseDir` был известен. Если документа на диске ещё нет, вернуть ошибку `asset_base_dir_missing`.
6. Вернуть URL и готовую Markdown-ссылку.

Permission:

```ts
type PluginPermission =
  | 'document:read'
  | 'document:write'
  | 'dialog'
  | 'export:html'
  | 'export:pdf'
  | 'assets:write';
```

## Plugin Manifest

Плагин использует минимальные существующие permissions:

```json
{
  "id": "markitdown-import",
  "name": "Import to Markdown",
  "version": "1.0.0",
  "description": "Конвертирует DOCX, XLSX, PDF, TXT, Markdown и CSV в Markdown и вставляет результат в документ.",
  "author": "Mivra Team",
  "entry": "index.js",
  "styles": "style.css",
  "permissions": ["document:read", "document:write", "dialog", "assets:write"],
  "apiVersion": 1
}
```

`document:read` нужен для формирования нового содержимого при вставке в конец или с разделителем. `document:write` нужен для применения результата. `dialog` нужен для UI. `assets:write` нужен для сохранения извлечённых изображений через S3 или локальную папку `assets/`.

## Поток Данных

1. Пользователь открывает пункт `Import to Markdown`.
2. Плагин показывает диалог с выбором файла, режимом применения и областью предпросмотра.
3. Пользователь выбирает файл.
4. Плагин определяет формат по имени файла и MIME type.
5. Плагин читает файл через browser `File` API.
6. Подходящий JS-конвертер возвращает Markdown и, если нашёл изображения, сохраняет их через `api.assets.saveBytes(...)`.
7. Плагин сохраняет результат в состоянии диалога и показывает предпросмотр.
8. Пользователь выбирает действие применения.
9. Плагин либо вызывает `api.document.setContent(nextContent)`, либо пишет результат в `navigator.clipboard`.

## Правила Применения

`Заменить документ`:

```text
convertedMarkdown
```

`Вставить в конец`:

```text
currentMarkdown

convertedMarkdown
```

`Вставить с заголовком и разделителем`:

```text
currentMarkdown

---

## Импортировано из source.docx

convertedMarkdown
```

Если текущий документ пустой, режимы вставки не добавляют лишние пустые строки в начало.

Плагин не применяет результат автоматически после конвертации. Применение всегда отдельное действие пользователя.

## Качество Конвертации

DOCX:

- сохранять заголовки, абзацы, списки, ссылки и простые таблицы;
- извлекать изображения, сохранять их через `api.assets.saveBytes(...)` и вставлять `![alt](url)`;
- если отдельное изображение не удалось сохранить, вставлять заметное текстовое предупреждение рядом с местом изображения и продолжать импорт остального документа;
- сложная верстка Word не является целью.

XLSX:

- каждый лист становится отдельным разделом `## <sheet name>`;
- диапазон данных листа превращается в Markdown-таблицу;
- пустые строки и колонки по краям таблицы отбрасываются;
- формулы выводятся как рассчитанные значения, если библиотека их отдаёт.

PDF:

- каждый текстовый PDF конвертируется постранично;
- страницы разделяются HTML-комментарием `<!-- page N -->` или заголовком, если это лучше читается в предпросмотре;
- сканы без текстового слоя показывают ошибку с объяснением, что OCR не входит в первый релиз;
- изображения PDF извлекаются best-effort, если выбранная JS-библиотека позволяет получить их как bytes;
- таблицы PDF не обещают точного восстановления.

TXT, MD, CSV:

- `.md` и `.markdown` вставляются как есть с нормализацией переносов строк;
- `.txt` вставляется как plain text;
- `.csv` превращается в Markdown-таблицу.

## Ошибки

Диалог показывает ошибки без изменения документа:

- формат не поддержан;
- файл не выбран;
- файл не удалось прочитать;
- конвертер выбросил ошибку;
- результат пустой;
- PDF не содержит текстового слоя;
- извлечённое изображение не удалось сохранить в S3 или `assets/`;
- текущий документ не сохранён, а S3 не готов, поэтому локальная папка `assets/` недоступна;
- буфер обмена недоступен.

Пользователь всегда может вручную скопировать Markdown из предпросмотра, если применение или clipboard недоступны.

## Безопасность

Плагин доверенный, как и остальные плагины API v1. При этом первый релиз не выполняет локальные процессы, не запускает сервер и не принимает пути к файлам от пользователя как строковые команды.

Все файлы читаются только через браузерный `File` API после явного выбора пользователя.

Файлы могут отправляться в сеть только через существующий S3 pipeline Mivra, если пользователь сам настроил S3 и подтвердил соединение. Без готового S3 изображения сохраняются локально в `assets/`.

## Сборка

Если выбранные JS-библиотеки требуют bundling, у плагина будет собственный build-процесс внутри `plugins/markitdown-import/`, аналогичный `plugins/export-pdf/`:

- исходники TypeScript/JavaScript находятся в `plugins/markitdown-import/src/`;
- сборка создаёт готовые `index.js`, `style.css` и `plugin.json`;
- готовая папка пакуется в `plugins/markitdown-import-1.0.0.mivraplugin`.

Корневой `npm run build:plugins` не меняется в первом релизе, чтобы не затрагивать ядро и существующий pipeline. Пакет плагина можно собирать отдельной командой из папки плагина.

## Тестирование

Тесты первого релиза:

- manifest содержит ожидаемые поля и permissions;
- `activate` регистрирует renderer-диалог и пункт меню;
- диалог рендерит выбор файла, режимы применения и кнопки;
- выбор неподдержанного формата показывает ошибку;
- `textToMarkdown` нормализует переносы строк;
- `csvToMarkdown` формирует Markdown-таблицу;
- `xlsxToMarkdown` формирует разделы по листам и таблицы;
- `pdfToMarkdown` возвращает постраничный Markdown для PDF с текстовым слоем;
- `api.assets.saveBytes` запрещён без permission `assets:write`;
- `api.assets.saveBytes` при готовом S3 вызывает `s3UploadBytes` и возвращает image Markdown;
- `api.assets.saveBytes` при неготовом S3 вызывает `saveLocalAssetBytes` и возвращает локальный URL;
- `api.assets.saveBytes` возвращает `asset_base_dir_missing`, если S3 не готов и документ ещё не сохранён;
- DOCX-конвертер сохраняет извлечённые изображения через assets API;
- `Заменить документ` вызывает `api.document.setContent(markdown)`;
- `Вставить в конец` сохраняет текущий документ и добавляет Markdown без лишних пустых строк;
- `Вставить с заголовком и разделителем` добавляет имя файла в заголовок;
- `Скопировать` вызывает `navigator.clipboard.writeText`;
- ошибка конвертации не меняет документ.

Проверки перед завершением реализации:

- `npm run test`;
- `npx tsc --noEmit`;
- отдельная сборка пакета `markitdown-import`;
- ручная проверка `.docx`, `.xlsx`, `.pdf`, `.txt`, `.md`, `.csv`;
- `graphify update .` после code-правок.

## Принятые Решения

Первый релиз делает не полный порт Microsoft MarkItDown, а JS-импортёр для нужных форматов Mivra.

Сервер, Python helper и автозапуск исключены из дизайна. Базовое приложение расширяется только в месте, где плагину нужен доступ к уже существующему asset pipeline.

Изображения DOCX являются обязательной частью первого релиза. Изображения PDF - best-effort, потому что качество извлечения зависит от возможностей выбранной JS-библиотеки и структуры конкретного PDF.
